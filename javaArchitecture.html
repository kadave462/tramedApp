0 The Big Picture

A Java project is a collection of plain text .java files. Before you can run them, three things must happen:

You write Maven builds JVM runs
┌────────────────┐ ┌────────────────┐ ┌────────────────┐ │ MyClass.java │ │ MyClass.class │ │ app.jar │ │ (plain text, │───►│ (bytecode, │───►│ (everything │───► Running App │ human-readable│ │ machine-safe) │ │ bundled) │ └────────────────┘ └────────────────┘ └────────────────┘
javac (compiler) mvn package java -jar

This whole process is managed by Maven. You just type mvn clean package and Maven does all three steps automatically.

Think of it like cooking: .java = raw ingredients, Maven = the chef who follows the recipe in pom.xml, .jar = the finished dish, JVM = the oven that heats (runs) it.

1 Java Source Files (.java)

Every piece of logic in a Java project lives in a .java file. Each file contains one class — a blueprint for doing something.

SparkBind's main files

C:\SparkBind\src\main\java\com\hirwa\
├── Main.java ← Entry point: connects to DB, schedules syncs ├── PharmacyDistributedJob.java ← Syncs products, patients, invoices every hour ├── core/ │ ├── database/ │ │ └── DerbyClient.java ← Opens a connection to the Derby database │ ├── api/ │ │ └── HirwaClient.java ← Sends HTTP POST requests to Hirwa API │ └── s3/ │ └── S3Uploader.java ← Uploads database backup to AWS S3

What a class looks like

package com.hirwa;              // like a folder address for this file

import java.sql.Connection;    // bring in JDBC Connection class
import java.sql.ResultSet;

public class DerbyClient {

    // A "field" — this object holds the DB connection
    private Connection connection;

    // A "method" — code that runs when called
    public Connection connect(String url) throws Exception {
        connection = DriverManager.getConnection(url);
        return connection;
    }
}
Package = folder for classes. com.hirwa means the file is stored at src/main/java/com/hirwa/. It prevents class name conflicts across libraries.

2 Maven + pom.xml — the Build System

Maven is the tool that manages your entire project: it downloads libraries, compiles code, and packages everything into a JAR.

Its config file is pom.xml — "Project Object Model". Think of it as the project's recipe card.

What pom.xml does (SparkBind example)

<!-- pom.xml — C:\SparkBind\pom.xml -->
<project>
  <!-- WHO this project is -->
  <groupId>com.hirwa</groupId>
  <artifactId>SparkBind</artifactId>
  <version>1.0-SNAPSHOT</version>

  <!-- SETTINGS: which Java version to use -->
  <properties>
    <maven.compiler.source>21</maven.compiler.source>
    <maven.compiler.target>21</maven.compiler.target>
  </properties>

  <!-- LIBRARIES to download (dependencies) -->
  <dependencies>
    <dependency>                  <!-- Apache Derby database driver -->
      <groupId>org.apache.derby</groupId>
      <artifactId>derbyclient</artifactId>
      <version>10.17.1.0</version>
    </dependency>
    <dependency>                  <!-- Jackson: converts Java → JSON -->
      <groupId>com.fasterxml.jackson.core</groupId>
      <artifactId>jackson-databind</artifactId>
    </dependency>
    <dependency>                  <!-- AWS SDK: upload to S3 -->
      <groupId>software.amazon.awssdk</groupId>
      <artifactId>s3</artifactId>
    </dependency>
  </dependencies>
</project>

Maven's lifecycle — what happens when you run mvn clean package

PhaseWhat Maven Does
cleanDeletes old target/ folder (removes stale compiled files)
validateChecks pom.xml is correct
compileReads all .java files → produces .class files in target/classes/
testRuns unit tests (we skip with -DskipTests)
packageBundles everything into target/SparkBind-1.0-SNAPSHOT.jar

Where Maven stores downloaded libraries

C:\Users\TRAMED\.m2\repository\ ← Maven's local "library shelf" ├── com\fasterxml\jackson\ ├── org\apache\derby\ └── software\amazon\awssdk\
First run: Maven downloads from internet (Maven Central). Next run: Maven reuses local cache (fast).
Maven Central is like the App Store for Java libraries. Every library in pom.xml is downloaded from there and stored locally in ~/.m2 so you don't download it again.

3 Compilation — .java → .class → .jar

Step 1: javac turns .java into .class (bytecode)

Main.java (text you can read)
↓ javac compiles
Main.class (bytecode — not text, not machine code, something in between)

Bytecode is NOT the final machine code. It runs on any computer that has Java installed — Windows, Linux, Mac — without recompiling. That's the famous Java promise: "Write Once, Run Anywhere."

Step 2: maven-shade-plugin bundles everything into a JAR

A JAR file (Java Archive) is just a ZIP file containing:

SparkBind-1.0-SNAPSHOT.jar (this IS a .zip — rename it to .zip to see inside)
├── com/hirwa/Main.class ← your compiled code ├── com/hirwa/DerbyClient.class ├── com/hirwa/PharmacyDistributedJob.class ├── org/apache/derby/ ... ← Derby library (bundled in) ├── com/fasterxml/jackson/ ... ← Jackson library (bundled in) ├── META-INF/MANIFEST.MF ← tells JVM which class is the entry point └── application.properties ← your config file
Why we exclude *.SF / *.DSA / *.RSA files from the JAR:
Some libraries come with digital signatures. When you bundle multiple signed libraries into one JAR, the signatures conflict and Java refuses to run the app with the error "Invalid signature file digest". We fixed this in pom.xml by telling maven-shade-plugin to strip those signature files.

Running the JAR

# The JVM reads MANIFEST.MF → finds Main-Class → starts there
java -jar C:\SparkBind\target\SparkBind-1.0-SNAPSHOT.jar

4 The JVM — Java Virtual Machine

The JVM is the engine that actually runs your .class bytecode. It sits between your code and the operating system.

SparkBind.jar ↓ ┌─────────────────────────────┐ │ JVM │ │ ┌─────────────────────┐ │ │ │ Class Loader │ │ ← reads .class files from the JAR │ │ Bytecode Verifier │ │ ← checks code is safe │ │ JIT Compiler │ │ ← converts bytecode → real CPU instructions │ │ Garbage Collector │ │ ← frees unused memory automatically │ └─────────────────────┘ │ └─────────────┬───────────────┘ ↓ Windows / Linux / macOS (hardware)

JAVA_HOME — why it matters

When you run Maven or a Java program, the system needs to know where Java is installed. That's what JAVA_HOME is: a variable pointing to your JDK folder.

# In PowerShell — tells Maven to use JDK 25
$env:JAVA_HOME = "C:\Program Files\Java\jdk-25.0.3"
$env:PATH = $env:JAVA_HOME + "\bin;" + $env:PATH
JDK (Java Development Kit)
The full toolkit: includes the compiler (javac), debugger, and the JVM. Needed by Maven to build your project.
JRE (Java Runtime Environment)
Just the JVM — no compiler. Enough to run an already-built JAR, but not to build it.

5 JDBC — How Java Talks to Databases

JDBC = Java Database Connectivity. It is a standard interface built into Java that lets your code talk to any database — Derby, MySQL, PostgreSQL, SQL Server — using the same code pattern.

Your Java Code └── DriverManager.getConnection(url) ↓ JDBC API (built into Java — java.sql.*) ↓ Database Driver ← a JAR from your pom.xml that knows the specific DB protocol ├── derbyclient.jar ← Derby protocol (DRDA over TCP port 1527) ├── postgresql.jar ← PostgreSQL protocol └── mssql-jdbc.jar ← SQL Server protocol (TDS over TCP port 1433) ↓ The actual Database (file on disk or network server)

JDBC URL — the database address

The URL tells JDBC where the database is. It looks like a web address but for databases:

// Pattern:
jdbc:protocol://host:port/database-name

// SparkBind connects to Ishyiga POS Derby database:
jdbc:derby://localhost:1527/SERVER1/TRAMEDNYARUGENGE
     ├──── ← driver/protocol      (Derby network client)
     │      ├──────── ← host     (this machine)
     │      │     ├── ← port     (Derby listens here)
     │      │         └───────────────────── ← database folder name

The JDBC pattern — always the same 4 steps

// 1. Open connection
Connection conn = DriverManager.getConnection("jdbc:derby://localhost:1527/SERVER1/TRAMEDNYARUGENGE");

// 2. Create a SQL statement
PreparedStatement stmt = conn.prepareStatement("SELECT * FROM products WHERE active = 1");

// 3. Execute and get results
ResultSet rs = stmt.executeQuery();

// 4. Loop through rows
while (rs.next()) {
    String name  = rs.getString("product_name");
    int    stock = rs.getInt("quantity");
    System.out.println(name + " — stock: " + stock);
}
JDBC does not care which database you use. You change the URL and the driver JAR — the code above stays identical whether you're talking to Derby, MySQL, or SQL Server.

6 Apache Derby — The Local Database

Apache Derby is a Java-only database. It comes bundled with Java itself and stores data as a folder on disk — not a single file.

Embedded Mode
The database runs inside your app's JVM. No network, no port.

URL: jdbc:derby:path/to/folder

Only one Java app can open it at a time. Ishyiga POS locks the main TRAMEDNYARUGENGE folder this way.
Network Mode
A Derby server opens the database and listens on port 1527. Other apps connect over TCP.

URL: jdbc:derby://host:1527/dbname

VSDC (EBM tax software) runs the Derby Network Server, making the database accessible over the network. SparkBind connects to this.

The database folder structure

E:\Ishyiga_tramed new Package\SERVER1\TRAMEDNYARUGENGE\
├── seg0\ ← actual table data (binary files) ├── log\ ← transaction log (used for crash recovery) ├── tmp\ ← temporary working space ├── db.lck ← lock file (exists while DB is running) └── service.properties ← Derby internal config
⚠ This FOLDER is the database. Never move or rename files inside it manually.

Port 1527 — the XSDB6 error story

We encountered error XSDB6 when SparkBind tried to open the database in embedded mode while VSDC already had it open in embedded mode. The fix was to connect in network mode to the copy VSDC serves:

WhatWho runs itMode
Port 1527 listenerVSDC (PID 9428, JDK 17)Derby Network Server
TRAMEDNYARUGENGE (locked)VSDC (embedded)Embedded — SparkBind cannot open this
SERVER1/TRAMEDNYARUGENGEVSDC (serves over network)Network — SparkBind connects here ✓

7 HttpClient — Sending Data to the Cloud

Apache HttpClient (the httpclient dependency in pom.xml) lets your Java code send HTTP requests — the same protocol your browser uses.

SparkBind uses it to POST pharmacy data to the Hirwa central API.

SparkBind (this machine) Hirwa API Server (cloud)
┌─────────────────────────┐ ┌─────────────────────────┐ │ 1. Build JSON payload │ │ 4. Receive data │ │ 2. Sign with HMAC key │──── POST ────►│ 5. Validate signature │ │ 3. Send via HttpClient │◄── 200 OK ────│ 6. Save to central DB │ └─────────────────────────┘ └─────────────────────────┘
Internet (port 443, HTTPS)

Code pattern

// Build the HTTP POST request
HttpPost request = new HttpPost("https://api.hirwa.rw/inventory/sync");
request.setHeader("Content-Type", "application/json");
request.setHeader("X-API-KEY", apiKey);
request.setHeader("X-HMAC-Signature", signature);   // security proof
request.setEntity(new StringEntity(jsonBody));

// Execute and read response
CloseableHttpClient client = HttpClients.createDefault();
HttpResponse response = client.execute(request);
int statusCode = response.getStatusLine().getStatusCode();  // 200 = OK

HMAC — the security signature

HMAC (Hash-based Message Authentication Code) proves the request came from a trusted source. It works like a seal on an envelope:

StepWhat happens
1. CombineTake the JSON payload + a secret key only you and Hirwa know
2. HashRun it through a math function (SHA-256) → get a unique fingerprint
3. SendInclude the fingerprint in the HTTP header
4. VerifyHirwa server does the same math — if fingerprints match, the request is genuine
The secret.key and hmac.algorithm fields in application.properties are still empty — we need these from the Hirwa team before SparkBind can successfully POST data.

8 Jackson — Turning Java Objects into JSON

Databases return rows and columns. APIs expect JSON. Jackson is the library that converts between Java objects and JSON text.

Java Object (in memory) JSON (text to send over HTTP)
Product { Jackson { name = "Paracetamol" ──── serialize ──► "name": "Paracetamol", stock = 150 "stock": 150, price = 500.0 ◄── deserialize ── "price": 500.0 } }

How it works in code

// 1. Define a Java class to represent the data
public class Product {
    public String name;
    public int    stock;
    public double price;
}

// 2. Create the Jackson converter
ObjectMapper mapper = new ObjectMapper();

// 3. Java → JSON (to send to API)
Product p = new Product();
p.name = "Paracetamol";  p.stock = 150;
String json = mapper.writeValueAsString(p);
// → {"name":"Paracetamol","stock":150,"price":0.0}

// 4. JSON → Java (to read API responses)
Product loaded = mapper.readValue(json, Product.class);
System.out.println(loaded.name);  // Paracetamol
Jackson reads the Java class field names and automatically maps them to JSON keys. No manual parsing needed — it does the heavy lifting.

9 AWS SDK — Uploading Backups to S3

AWS S3 (Simple Storage Service) is Amazon's cloud file storage. SparkBind uses it to upload a ZIP of the Derby database folder every 12 hours as a cloud backup.

SparkBind (scheduled task) 1. ZIP the Derby folder → backup.zip 2. Use AWS SDK to authenticate with AWS 3. Upload backup.zip to S3 bucket ↓ Internet (HTTPS) Amazon AWS S3 └── bucket: hirwa-pharmacy-backups/ └── TRAMEDNYARUGENGE/ └── 2026-06-24_02-00.zip

How the AWS SDK works

// 1. Create S3 client (reads AWS credentials from env variables)
S3Client s3 = S3Client.builder()
    .region(Region.of("us-east-1"))
    .build();

// 2. Upload a file
PutObjectRequest req = PutObjectRequest.builder()
    .bucket("hirwa-pharmacy-backups")
    .key("TRAMEDNYARUGENGE/backup.zip")
    .build();

s3.putObject(req, Path.of("C:/backups/backup.zip"));

The AWS SDK needs credentials (AWS Access Key ID + Secret) configured in application.properties or as environment variables (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY).
